What is MIRAGE? (The Big Picture)

MIRAGE stands for Modular Intelligence for Reconnaissance Analysis and Gathering Engine. It is a honeypot system — a cybersecurity tool designed to attract, trap, and study real hackers and automated bots on the internet.

🧠 Think of it like this

Imagine you are a police detective and you want to catch thieves. Instead of waiting for them to rob a real bank, you set up a fake bank that looks completely real. Thieves walk in thinking they're robbing a real bank, but every step they take is being recorded by hidden cameras. Afterwards, you study the footage to learn their techniques and catch them. MIRAGE is that fake bank, but for the internet.

What does MIRAGE actually do?

  1. Sets a trap — It runs a fake SSH server (like a fake computer login) and a fake website, both exposed to the internet.
  2. Attracts attackers — Hackers and bots that scan the internet discover your server and try to break in.
  3. Records everything — Every login attempt, every command they type, every page they visit is silently captured and stored in a database.
  4. Analyzes the attacks — A math-based engine groups similar attackers together (clustering) and labels their actions using a global cybersecurity framework (MITRE ATT&CK).
  5. Displays the results — A premium web dashboard shows you charts, heatmaps, and live feeds of everything happening.
💡 Why does this matter? Honeypots are used by real cybersecurity companies, governments, and researchers worldwide. By building and running one, you are demonstrating real-world security engineering skills.

Core Concepts You Need to Know

🍯 Honeypot

A honeypot is a deliberately vulnerable computer system set up to attract attackers. It looks like a real target (a server, a website, etc.), but it's actually a trap. Everything the attacker does is monitored and logged. Honeypots are used to study attack techniques, collect malware samples, and improve defenses.

🔐 SSH (Secure Shell)

SSH is the most common way to remotely log into and control a Linux server over the internet. You type commands in a terminal (like ls, cat /etc/passwd), and the server executes them. Attackers use SSH to break into servers by guessing usernames and passwords (brute-force attacks). Our Cowrie honeypot pretends to be an SSH server so attackers think they've "broken in."

🌐 HTTP / Web Application

HTTP is the protocol your web browser uses. When you visit a website, your browser sends HTTP requests (like GET /index.html) and the server sends back the page. Web scanners are bots that automatically visit thousands of websites looking for known vulnerabilities like exposed config files (/.env) or admin panels (/admin). Our Flask decoy is a fake website designed to attract these scanners.

🐳 Docker & Containers

Docker is a tool that packages an application and all its dependencies into an isolated "container." Think of it like a lightweight virtual machine. Each MIRAGE service (Cowrie, Decoy, Dashboard, Ingestion) runs in its own container, isolated from each other. Docker Compose is a tool that starts all containers together with a single command: docker compose up -d.

🤖 Bot vs. Human Attacker

A bot is an automated program that scans the internet and attacks servers without human involvement. Bots type commands very quickly (100-500ms between each command) and follow the exact same pattern every time. A human attacker is a real person typing commands manually — their timing is irregular (1-15 seconds between commands) and they explore more creatively. MIRAGE's analytics engine can tell the difference.

Layer 1: Cowrie SSH Honeypot

Cowrie is an open-source SSH honeypot. It pretends to be a real Linux server. When an attacker connects via SSH and types commands, Cowrie:

  • Accepts common passwords like root/admin, root/123456
  • Presents a fake Linux shell that looks real but isn't
  • Lets attackers "run" commands like ls, cat /etc/passwd, wget
  • Records every single thing they type into a JSON log file
  • Can even capture malware files that attackers try to download
🧠 Real-world analogy

Cowrie is like a fake ATM machine set up by police. It looks real, it "accepts" cards, but instead of dispensing money, it silently photographs the thief and records everything they do.

What does Cowrie's JSON log look like?

// A real attacker trying to log in: {"eventid": "cowrie.login.success", "username": "root", "password": "admin", "src_ip": "185.220.101.45", "timestamp": "2026-07-09T15:30:00"} // The attacker typing a command: {"eventid": "cowrie.command.input", "input": "cat /etc/passwd", "src_ip": "185.220.101.45", "session": "abc123", "timestamp": "2026-07-09T15:30:05"}

Port: Cowrie listens on port 2222. When an attacker types ssh root@your-server -p 2222, they connect to Cowrie instead of a real server.

Layer 2: Flask HTTP Decoy

The Flask Decoy is a deliberately vulnerable web application that we built ourselves. It mimics a real company's internal web app, complete with intentional security bugs that attract hackers and scanners.

The 3 Intentional Vulnerabilities

BugWhat It DoesWhy Attackers Care
JWT Secret Leak The /debug/config endpoint exposes the secret key used to sign authentication tokens If an attacker finds this, they can forge admin tokens and impersonate any user
Auth Bypass The login page accepts admin/admin and can be bypassed by adding role=admin to the request Lets attackers gain admin access without knowing the real password
Permissive CORS Every response includes Access-Control-Allow-Origin: * Allows any website in the world to make requests to this server (a real security flaw)

Bait Endpoints (Attracting Scanners)

URL PathWhat It ReturnsWhy Scanners Look for It
/.envFake credentials fileMany web apps accidentally expose their .env file with database passwords
/.git/configFake git configExposed git repos let attackers download source code
/api/usersFake user list with emailsAttackers harvest email addresses for phishing
/debug/configFake config with JWT secretDebug endpoints often leak sensitive information
/robots.txtLists "hidden" pathsScanners read this to find interesting URLs to attack

How does it log everything?

The middleware.py file hooks into Flask's request pipeline. Before every single HTTP request is processed, the middleware silently captures the source IP, the request method and path (e.g., GET /.env), and the timestamp — then writes it all into the database as a "command" within an attacker's session.

Layer 3: Ingestion Engine

The Ingestion Engine (cowrie_watcher.py) is a long-running background process that acts as a bridge between Cowrie's JSON log file and the MIRAGE database.

What does it do?

  1. Watches the Cowrie log file (cowrie.json) for new lines being added
  2. Parses each JSON event (login attempt, command typed, session started/ended)
  3. Normalizes the data into clean rows that match our database schema
  4. Writes the normalized data into the SQLite database
🧠 Real-world analogy

Think of the Ingestion Engine as a court stenographer. The Cowrie honeypot is the courtroom where things happen, and the stenographer sits in the corner silently transcribing every word that is spoken into an organized, searchable document (the database).

Event types it handles

Cowrie EventWhat It MeansDatabase Action
cowrie.session.connectAttacker connected to the honeypotCreates a new session row + upserts sources
cowrie.login.successAttacker guessed the right passwordUpdates the session
cowrie.login.failedAttacker guessed wrongLogs the attempt
cowrie.command.inputAttacker typed a commandInserts into commands table
cowrie.session.closedAttacker disconnectedUpdates session end_time & duration

Layer 4: The Database

MIRAGE uses SQLite — a lightweight database that stores all its data in a single file called mirage.db. No external database server needed.

commandsEvery command typed by every attacker"cat /etc/passwd" — typed 5.2 seconds after previous command

ℹ️ Why SQLite? The PRD specifically requires SQLite because it's a single file (easy to deploy), needs no server process, and is perfect for a self-contained honeypot system. For a production system handling thousands of concurrent attacks, you'd typically upgrade to PostgreSQL.

Layer 5: Analytics Engine

The Analytics Engine is the "brain" of MIRAGE. It takes raw captured data and turns it into actionable intelligence. It has three components:

⚡ Zero AI / Zero LLMs: The entire analytics engine uses only math and regex patterns. There are absolutely no AI models, no ChatGPT calls, no machine learning models that need training. This is a deliberate design choice — the system must work reliably without internet access or API keys.

Component 1: DBSCAN Clustering

File: analytics/clustering.py

Purpose: Groups similar attacker sessions into "campaigns."

How it works:

  1. For each session, it extracts 19 numerical features like: total commands, unique commands, session duration, average typing speed, percentage of recon commands, etc.
  2. It feeds these feature vectors into the DBSCAN algorithm (explained in Section 13).
  3. DBSCAN groups sessions that have similar feature profiles. For example, 20 sessions that all type the same 5 commands with 200ms delays → grouped as a single botnet campaign.
  4. Sessions that don't fit any group are marked as "noise" (outliers) — these are often the most interesting because they might be manual human hackers.

Component 2: ATT&CK Tagger

File: analytics/attack_tagger.py

Purpose: Labels each command with a standardized attack technique from the MITRE ATT&CK framework (explained in Section 12).

How it works:

  1. Loads 30 regex (regular expression) patterns from the database.
  2. For every command in every session, it checks if the command matches any pattern.
  3. If cat /etc/passwd matches the pattern for "T1003.008 — OS Credential Dumping," it creates a link in the session_techniques table.

Component 3: Bot Detector

File: analytics/bot_detector.py

Purpose: Classifies each session as "bot" or "human" based on typing patterns.

How it works:

  • Bots type at very consistent speeds (low variance in time between commands)
  • Humans type at irregular speeds (high variance — sometimes fast, sometimes slow while they think)
  • The detector calculates the coefficient of variation (standard deviation ÷ mean) of the inter-command timing. High variation = human. Low variation = bot.

What is MITRE ATT&CK?

MITRE ATT&CK (Adversarial Tactics, Techniques, and Common Knowledge) is a globally recognized knowledge base of cyberattack techniques. It was created by the MITRE Corporation and is used by every major cybersecurity company in the world.

🧠 Real-world analogy

Think of MITRE ATT&CK like the Periodic Table of Elements, but for hacking. Just like every chemical element has a unique symbol (H, He, Li...), every known attack technique has a unique ID (T1003, T1082, T1059...). When a security analyst sees "T1003.008," they instantly know it means "the attacker tried to dump credentials from /etc/passwd."

Techniques MIRAGE detects (examples from the 30 loaded)

IDNameWhat the Attacker Does
T1003.008OS Credential DumpingReads /etc/passwd or /etc/shadow to steal passwords
T1082System Info DiscoveryRuns uname -a to learn about the system
T1105Ingress Tool TransferUses wget or curl to download malware
T1496Resource HijackingInstalls crypto miners like xmrig
T1070.003Clear Command HistoryRuns history -c to cover their tracks
T1059.004Unix ShellSpawns a shell with bash -i or python -c

How does MIRAGE detect these? Each technique has a regex pattern stored in the technique_signatures database table. When a command like cat /etc/passwd matches the regex (?:cat|head|tail).*\/etc\/(?:passwd|shadow), the system tags that session with technique T1003.008.

What is DBSCAN Clustering?

DBSCAN (Density-Based Spatial Clustering of Applications with Noise) is a mathematical algorithm that automatically finds groups of similar items in data — without you having to tell it how many groups to look for.

🧠 Real-world analogy

Imagine you throw 100 marbles on a table. Some marbles land close together in small piles, and a few land far away from everything else. DBSCAN is like an algorithm that identifies each pile as a "cluster" and labels the loners as "noise." You don't have to tell it how many piles there are — it figures it out by itself based on how close the marbles are to each other.

How MIRAGE uses DBSCAN

  1. Feature Extraction: For each session, we calculate 19 numbers that describe the session's behavior (how many commands, how fast they typed, what percentage were recon commands, etc.)
  2. Normalization: We scale all numbers to the same range so no single feature dominates
  3. DBSCAN: The algorithm looks at these 19-dimensional "points" and finds clusters of sessions that behave similarly
  4. Result: Each cluster becomes a "campaign" — e.g., "20 sessions from different IPs that all ran the same 5 commands with identical timing are probably the same botnet"

Why DBSCAN and not K-Means?

FeatureDBSCAN ✅K-Means ❌
Requires you to specify number of clustersNo (figures it out)Yes (you must guess)
Handles outliersYes (labels them as noise)No (forces them into a cluster)
Finds non-spherical clustersYesNo

Security & Safety Considerations

🔴 CRITICAL: MIRAGE is intentionally vulnerable. The Flask decoy has real security bugs. The SSH honeypot accepts attacker connections. This system MUST run on an isolated cloud server (like AWS EC2), never on your personal computer or home network.

Why isolation matters

  • The system is designed to attract real hackers. If an attacker somehow escapes the Docker container, they must NOT have access to your personal files, home network, or other devices.
  • A cloud VM is physically separated from your home. If it gets compromised, you just delete the VM from the AWS console — zero risk to you.
  • Never store real credentials, personal data, or sensitive information on the same server as the honeypot.

What attackers CAN and CANNOT do

Can ✅Cannot ❌
Connect to the fake SSH shellAccess the real operating system
Type commands in Cowrie's sandboxExecute real commands on the server
Visit the decoy websiteAccess the dashboard (different port)
Download files into Cowrie's fake filesystemActually install software on the server

How to Present This Project

When presenting MIRAGE for your internship or portfolio, here are the key points to emphasize:

Opening Statement

"MIRAGE is a deception intelligence platform that I built to study real-world cyberattacks. It deploys a multi-protocol honeypot on an isolated cloud server, captures live attacker sessions, and uses mathematical clustering to automatically group attackers into campaigns and map their behavior to the MITRE ATT&CK framework — all without using any AI or machine learning models."

Key Technical Highlights to Mention

  • Multi-protocol capture: Both SSH (Cowrie) and HTTP (Flask decoy) attack surfaces
  • Real attacker data: Not simulated — actual bots and hackers from the internet
  • Rule-based analysis: DBSCAN clustering + regex-based ATT&CK technique tagging (no AI dependency)
  • Production-grade deployment: Docker containerized, running on AWS EC2
  • Live dashboard: Real-time command feed, interactive charts, ATT&CK heatmap
  • 19-feature behavioral profiling: Each session is mathematically profiled for clustering
  • Bot vs. human classification: Statistical analysis of command timing variance

Questions an interviewer might ask

QuestionYour Answer
"Why not use ML for classification?""The PRD requires zero AI dependency. Rule-based systems are more predictable, auditable, and don't require training data or API keys."
"Why SQLite instead of PostgreSQL?""For a single-node honeypot, SQLite provides zero-config deployment and sufficient performance. It would be upgraded for a distributed system."
"How do you tell bots from humans?""I analyze the coefficient of variation in inter-command timing. Bots have near-zero variance; humans have high variance."
"What's DBSCAN?""A density-based clustering algorithm that finds groups without needing a predefined number of clusters, and naturally identifies outliers as noise."
"Is this legal?""Yes. Running a honeypot on your own server is legal. We only capture data from attackers who voluntarily connect to our system."
💡 Pro tip: When presenting, have the live dashboard open and show real captured data. Nothing impresses interviewers more than a system that is actively catching real attackers while you present it.
